# Input Files for a Classification Project

This document describes the per-sample files that make up an AmpliconSuite
classification project, which ones the Aggregator needs, and the naming
patterns it uses to recognize them.

## Where each file comes from

| Tag | Source |
|-----|--------|
| **[AA]** | AmpliconArchitect output |
| **[AC]** | AmpliconClassifier-generated |
| **[ASP]** | AmpliconSuite-pipeline.py generated (some derived from CNVKit outputs) |

## A. Essential

A classification project has two halves: the **[AA]** inputs required to
(re-)run classification, and the **[AC]** outputs that *are* the classification.

### Classification inputs — required to (re-)run AmpliconClassifier
| File | Source | Notes |
|------|--------|-------|
| `[sample]_amplicon[N]_graph.txt` | **[AA]** | Breakpoint graph; primary AC input |
| `[sample]_amplicon[N]_cycles.txt` | **[AA]** | Cycles/paths; primary AC input |
| `[sample]_summary.txt` | **[AA]** | Amplicon summary; required to re-classify, and used to validate an AA results directory (header must start with `#Amplicons`) |

### Classification output — the result itself
| File | Source | Notes |
|------|--------|-------|
| `[prefix]_result_table.tsv` | **[AC]** | Authoritative source of truth — the Aggregator builds its sample registry from this file |
| `[prefix]_amplicon_classification_profiles.tsv` | **[AC]** | A directory is only recognized as a classification dir when *both* this and the result table are present |

### What `_result_table.tsv` holds

The result table is the master index of AmpliconClassifier's per-feature
output — one row per (sample, amplicon, feature) — bundling several AC
outputs into a single table:

- **Calls / annotations AC computed:** `Classification`, `Feature ID`,
  `Location`, `Oncogenes`, `All genes`, `Complexity score`,
  `Captured interval length`, `Feature median copy number`,
  `Feature maximum copy number`, `Filter flag`
- **Provenance carried through:** `Reference version`, `Tissue of origin`,
  `Sample type`
- **Pointers to other files:** `Feature BED file`, `CNV BED file`,
  `AA PNG file`, `AA PDF file`, `AA cycles file`, `AA graph file`,
  `Run metadata JSON`, `Sample metadata JSON`

Because it references nearly every other file in the project, the
standalone AC tables (`_gene_list.tsv`, `_feature_basic_properties.tsv`,
`_feature_entropy.tsv`, `_ecDNA_counts.tsv`, `_ecDNA_context_calls.tsv`)
are essentially the same per-feature data broken back out into separate files.

## B. Strongly recommended

Not required to classify, but omitting them leaves parts of the
AmpliconRepository.org display blank.

| File | Source | Notes |
|------|--------|-------|
| `[sample]_amplicon[N].png` | **[AA]** | Per-amplicon visualization. Without the `.png`, elements of the AmpliconRepository.org site render blank |
| `[sample]_amplicon[N].pdf` | **[AA]** | Per-amplicon visualization (print/vector) |
| `[sample]_CNV_CALLS.bed` | **[ASP]** | Copy-number calls, generated by AmpliconSuite-pipeline.py from the CNVKit `.cns` file; also drives site elements that otherwise go blank |

## C. Optional

Useful for reproducibility or convenience, but re-derivable from the
essentials or not consumed downstream.

| File | Source | Notes |
|------|--------|-------|
| `[sample]_AA_CNV_SEEDS.bed` | **[AA]** | Seed intervals AA was launched from (reproducibility only) |
| `[sample]_run_metadata.json` | **[ASP]** | Reference genome, tool versions. Note: the Aggregator exits fatally if the reference genome is inconsistent across samples |
| `[sample]_sample_metadata.json` | **[ASP]** | Tissue/sample type and related provenance |
| `[sample]_finish_flag.txt` | **[ASP]** | Run-completion marker — useful for detecting whether a run crashed before finishing |
| `[prefix]_gene_list.tsv` | **[AC]** | Derived per-feature table |
| `[prefix]_feature_basic_properties.tsv` | **[AC]** | Derived per-feature table |
| `[prefix]_feature_entropy.tsv` | **[AC]** | Derived per-feature table |
| `[prefix]_ecDNA_counts.tsv` | **[AC]** | Derived per-feature table |
| `[prefix]_ecDNA_context_calls.tsv` | **[AC]** | Derived per-feature table |
| `[prefix]_annotated_cycles_files/` | **[AC]** | Derived per-feature directory |
| `[prefix]_classification_bed_files/` | **[AC]** | Derived per-feature directory |
| `[prefix]_SV_summaries/` | **[AC]** | Derived per-feature directory |

## D. Nice-to-haves

Run diagnostics. The Aggregator carries these through when present, but
nothing downstream consumes them — they don't feed classification or the
AmpliconRepository.org display.

| File | Source | Notes |
|------|--------|-------|
| `[sample].log` | **[AA]** | AmpliconArchitect run log |
| `[sample]_timing_log.txt` | **[ASP]** | Per-stage timing |
| `[sample].cns` | CNVKit | Segmented copy-number calls in log2 ratio (inside the `_cnvkit_output` directory); `_CNV_CALLS.bed` is derived from this, converted to real (linear) copy numbers |

## Naming patterns

The Aggregator identifies files by prefix and suffix, so consistent naming
matters more than directory layout.

**Sample names** are derived from directory/file prefixes — e.g. `SAMPLE`
from `SAMPLE_AA_results/` or `SAMPLE_summary.txt`.

**Directory suffixes** used for type identification:

| Suffix | Contains |
|--------|----------|
| `_AA_results` | AmpliconArchitect per-sample output |
| `_cnvkit_output` / `_cnvkit_outputs` | CNVKit output |
| `_classification` | AmpliconClassifier output |

**Per-amplicon files** follow `[sample]_amplicon[N]<suffix>`, where `[N]`
is the amplicon number and `<suffix>` is one of `.png`, `.pdf`,
`_cycles.txt`, `_graph.txt`:

```
GBM39_amplicon1_graph.txt
GBM39_amplicon1_cycles.txt
GBM39_amplicon1.png
GBM39_amplicon1.pdf
```

**Classification (AC) outputs** are prefixed with a shared project/run
prefix and use the suffixes listed in the tables above, e.g.:

```
GBM39_result_table.tsv
GBM39_amplicon_classification_profiles.tsv
```

The result table and profiles file must share the same prefix to be paired.
